Ниже план в формате, пригодном для раздачи подагентам и последовательного слияния через отдельные worktree.

Сейчас переход на MobX выполнен частично: появился `RootStore`, но он содержит только один активный `ChatStore`; параллельно остаются Zustand-хранилища и старый `useChatThread`. Два маршрута чата используют разные реализации, а Jest глобально подменяет MobX заглушкой, поэтому тесты не подтверждают реальную реактивность. На backend worktree и путь Pi-сессии уже создаются, но основной chat runtime запускается с глобальным `PI_CWD`, а клиентский API заметно шире зарегистрированных Hono-маршрутов. ([GitHub][1])

# Целевой milestone

Ближайшая крупная цель должна быть такой:

> Пользователь запускает две параллельные implementation-задачи из Web/Android/iOS. Каждая работает в собственном worktree и собственной постоянной Pi-сессии. Обе продолжают работу после перезапуска backend. С телефона можно видеть streaming, tool calls, отправлять steer/follow-up/abort. После завершения отображается diff, создаётся checkpoint, одна задача сливается в основной репозиторий, а вторая корректно определяется как stale.

Пока этот сценарий не проходит целиком, новые экраны и интеграции лучше не считать приоритетом.

## F03. Исправить модель realtime cursor

Сейчас клиентский reducer считает события устаревшими через лексикографическое сравнение `event.id`, а сервер генерирует собственный timestamp/counter ID. Это лучше заменить явным числовым cursor, особенно для восстановления после рестартов. ([GitHub][3])

### Реализация

Добавить в событие:

```ts
type RealtimeEnvelope = {
  id: string;
  sequence: number;
  stream: "project" | "chat" | "task";
  streamId: string;
  // ...
};
```

Правила:

```text
id       = уникальный ключ для дедупликации
sequence = cursor для порядка и resume
```

SSE:

```http
GET /api/chats/:id/events?afterSequence=1042
```

Клиент хранит cursor отдельно для каждого stream.

### Проверка

* События с одинаковым `id` не применяются повторно.
* События с меньшим `sequence` игнорируются.
* После рестарта API sequence продолжает расти.
* Reconnect возвращает только события после cursor.
* Одновременные project/chat/task streams не смешивают cursor.

# Этап 2. Task-aware Pi runtime

## R01. Зафиксировать domain model

Ввести четыре разные сущности:

```text
Chat
  пользовательская ветка разговора

Task
  изолированная работа с filesystem и git branch

PiSession
  долгоживущий контекст Pi

Run
  один выполняющийся turn/task operation
```

Ограничения:

```text
- один PiSession имеет не более одного active Run;
- один writable Task владеет одним worktree;
- много Tasks могут работать параллельно;
- Chat может быть discussion без Task;
- implementation Chat имеет active Task;
- orchestration Chat может следить за несколькими Tasks.
```

### Проверка

Контракты и DB не используют `task`, `run` и `session` как взаимозаменяемые понятия.

```ts
type TaskRuntimeContext = {
```

## R03. Явный lifecycle run

### Реализация

Команда prompt должна иметь два этапа:

```ts
type RunHandle = {
  accepted: Promise<void>;
  completed: Promise<RunResult>;
};
```

`accepted` означает, что Pi принял RPC-команду.
`completed` означает:

```text
agent_end
run.aborted
run.error
process exit
timeout
```

Текущий Pi adapter отдельно отмечает, что `prompt()` возвращается после RPC acknowledgement и не ждёт `agent_end`, хотя `PiRpcClient` уже имеет `waitForIdle`. ([GitHub][7])

### Проверка

* Task не получает `needs_review` до `agent_end`.
* Lock не освобождается после одного только RPC response.
* Tool event после RPC response не теряется.
* Process exit переводит run в `failed`.
* Timeout создаёт terminal event.

## R05. Persistent Pi sessions

### Реализация

DB уже содержит поля `path`, `cwd`, active leaf, cursor и lock. Нужно сделать их частью основного runtime flow, а не отдельной подсистемой. ([GitHub][8])

```text
1. создать PiSession record;
2. создать директорию session;
3. связать chat/task/session;
4. запустить Pi с этим session path;
5. сохранять active leaf и import cursor.
```

При старте API:

```text
- проверить незавершённые runs;
- очистить expired locks;
- восстановить session metadata;
- не запускать Pi до первого обращения или background resume.
```

### Проверка

```text
1. Начать чат.
2. Завершить первый prompt.
3. Перезапустить API.
4. Открыть тот же чат.
5. Отправить второй prompt.
6. Pi видит контекст первого prompt.
```

# Этап 3. Worktree, checkpoints, fork и merge

## G01. Модель основного repository checkout

Для MVP:

```text
repoPath
  управляемый integration checkout

runtimeStatePath/worktrees/<taskId>
  task worktrees
```

Правила:

```text
- Pi не пишет в integration checkout;
- merge выполняется под project-level mutex;
- перед merge integration checkout обязан быть clean;
- пользовательские незакоммиченные изменения блокируют merge.
```

В перспективе лучше перейти к:

```text
bare/common git dir
integration worktree
task worktrees
```

Текущий merge service выполняет `git checkout` target/branch прямо в `repoPath`, что опасно при linked worktrees и параллельной работе. ([GitHub][9])

### Проверка

* Запущенная task не меняет `git status` integration checkout.
* Одновременно выполняется только одна merge/rebase операция на проект.
* Dirty integration checkout приводит к понятной ошибке, а не потере файлов.

---

## G02. Автоматический checkpoint после turn

### Реализация

После успешного `agent_end`:

```text
1. git status;
2. git add -A;
3. commit в task branch;
4. сохранить beforeSha/afterSha;
5. связать checkpoint с Pi entry;
6. сохранить patch;
7. обновить task currentHeadSha.
```

Patch хранить вне worktree:

```text
runtimeStatePath/
  checkpoints/
    <taskId>/
      <checkpointId>.patch
```

Сейчас checkpoint service создаёт `.checkpoints` непосредственно внутри task worktree. Это нужно убрать, чтобы служебные patches не попадали в последующие commits и diff. ([GitHub][10])

### Проверка

* После turn рабочее дерево clean.
* Patch отсутствует в `git status`.
* Checkpoint связан с SHA и Pi entry.
* Empty turn допускается только по явной политике.
* Сбой commit не переводит task в `needs_review`.

---

## G03. Diff и changedFiles

### Реализация

Вычислять:

```text
task baseSha..currentHeadSha
checkpoint beforeSha..afterSha
task branch..merge target
```

Поддержать:

```text
added
modified
deleted
renamed
binary
```

### Проверка

* `changedFiles` в task соответствует Git.
* Diff после checkpoint и diff всей task различаются корректно.
* Binary file не ломает endpoint.
* Path traversal невозможен.

---

## G04. Fork task

### Реализация

Fork разрешён только от стабильного checkpoint.

```text
Task A checkpoint C2
  ↓
Task B
  branch starts at C2.afterSha
  own worktree
  own Pi session
  session forked from C2.piEntryId
```

### Проверка

* SHA новой ветки равен checkpoint SHA.
* Изменения A после fork не появляются в B.
* Изменения B не появляются в A.
* Pi-сессии имеют общую историю до fork point и разные продолжения.
* Fork активного незавершённого toolcall запрещён или сначала создаёт stable checkpoint.

---

## G05. Rollback

Два действия:

```text
Restore files in current task
Fork new task from checkpoint
```

Безопасный default:

```text
Fork new task from checkpoint
```

Destructive restore требует отдельного подтверждения.

### Проверка

* Старые commits не удаляются.
* Исходная task остаётся доступной.
* Новая task стартует с выбранного checkpoint.
* Pi context и filesystem соответствуют одной точке истории.

---

## G06. Rebase и stale detection

### Реализация

После merge другой task:

```text
- пересчитать behind target;
- idle task перевести в stale;
- активную task не rebase автоматически;
- предложить action Rebase from main.
```

Rebase выполняется в task worktree, а не через checkout task branch в integration checkout.

### Проверка

Сценарий:

```text
A и B стартуют от M0
A merge → M1
B становится stale
B rebase:
  либо успешно продолжает от M1
  либо получает merge_conflict
```

При конфликте файлы и conflict markers доступны в B worktree.

---

## G07. Merge

### Реализация

Default:

```text
Squash merge
```

Workflow:

```text
1. task idle/needs_review;
2. final checkpoint;
3. checks;
4. integration checkout clean;
5. stale check;
6. merge;
7. task → merged;
8. остальные task пересчитываются на stale;
9. worktree архивируется, но не удаляется немедленно.
```

### Проверка

* Main получает один итоговый commit.
* Checkpoint commits не засоряют main.
* Повторный merge не создаёт второй commit.
* При конфликте task получает `merge_conflict`.
* Ошибка merge оставляет integration checkout в восстановленном состоянии.

# Этап 4. API parity и решение по oRPC

## A01. Реестр API

Создать machine-readable registry:

```ts
type ApiOperation = {
  id: string;
  method: string;
  path: string;
  requestSchema?: ZodSchema;
  responseSchema: ZodSchema;
  implemented: boolean;
};
```

### Проверка

Каждый публичный метод `ApiClient` имеет operation ID и backend route.

---

## A02. Parity test

### Проверка

Автотест сравнивает:

```text
ApiClient operations
vs
registered Hono operations
```

Тест падает, если:

* метод клиента не имеет route;
* route не имеет response schema;
* path или method отличаются;
* endpoint возвращает response, не проходящий Zod parse.

## A03. Реализовать endpoints основного vertical slice

Приоритет:

```text

diff

```

После них:

```text

MCP
```

### Проверка

Все методы первого блока проходят интеграционные Hono-тесты без 404 и без mock response.

---

## A04. Решение по oRPC

Не делать полный переход на oRPC одновременно с MobX и runtime.

После работающего core milestone:

```text
1. реализовать один изолированный domain через oRPC;
2. сравнить объём contracts/client/server code;
3. проверить React Native compatibility;
4. проверить OpenAPI generation;
5. оставить SSE raw Hono endpoints.
```

Кандидат для эксперимента:

```text
providers
```

не `messages` и не realtime.

Критерий принятия oRPC:

```text
- меньше ручного client code;
- нет дублирования Zod;
- нормальная обработка ошибок;
- не усложняет native bundling;
- SSE остаётся независимым.
```

# Этап 8. Безопасность и VPS deployment

Даже при одном пользователе backend на VPS не должен оставаться полностью открытым.

## S01. Authentication

```text

```

### Проверка

## S02. Network policy

* Production CORS allowlist вместо `origin: '*'`.
* Поддержка Tailscale hostname/IP.
* HTTP только внутри tailnet либо HTTPS через reverse proxy.
* Ограничение body size.
* Rate limit для login/pairing/package resolve.

Текущий Hono app разрешает wildcard CORS, что допустимо для ранней разработки, но не для открытого VPS. ([GitHub][11])

# Обязательные integration gates

## Gate 1 — MobX завершён

```text
- нет импортов Zustand;
- нет глобальных mock MobX;
- один ChatScreen;
- несколько ChatSessionStore;
- background updates работают;
- realtime reconnect проходит.
```

## Gate 2 — одна настоящая task

```text
- создан worktree;
- Pi cwd равен worktree;
- Pi session постоянна;
- run заканчивается только после agent_end;
- checkpoint создаётся;
- diff показывается.
```

## Gate 3 — параллельность

```text
- две tasks работают одновременно;
- разные worktrees;
- разные Pi sessions;
- разные runs;
- main checkout не изменяется агентами;
- steering одной task не влияет на другую.
```

## Gate 4 — fork и merge

```text
- fork от checkpoint;
- ветки расходятся;
- A merge в main;
- B становится stale;
- B rebase или conflict;
- история и файлы не теряются.
```

## Gate 5 — восстановление

```text
- backend принудительно перезапущен;
- running task получает корректный recovered/failed status;
- завершённая Pi session продолжается;
- SSE cursor не дублирует историю;
- locks восстанавливаются.
```

# Definition of Done для любой задачи

Подагент не должен отмечать задачу выполненной, пока нет:

```text
1. Типизированной реализации.
2. Unit-тестов основной логики.
3. Интеграционного теста публичного контракта.
4. Проверки ошибки и отмены.
5. Отсутствия mock в production path.
6. Обновлённой документации.
7. Отчёта с выполненными командами.
8. Подтверждения, что изменения ограничены allowed_paths.
9. Зелёных typecheck/test/lint.
10. Слияния в integration branch и повторного прогона тестов.
```

# Первый практический пакет работ

Начать стоит с этих пяти карточек:

```text
F00 — reproducible dependencies and CI

M01 — RootStoreProvider and dependency injection
R01 — Chat/Task/Run/PiSession domain model
```
