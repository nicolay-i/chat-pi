Принято. После этих уточнений основная продуктовая модель стала достаточно однозначной для реализации. Ниже фиксирую её как нормативную спецификацию.

# 1. Каноническая модель

## 1.1. Выполнение только на VPS

В текущей версии на VPS находятся:

```text
Hono backend
SQLite/PostgreSQL
основное Git-репо
task worktrees
Pi processes
Pi session files
sandbox/process supervisors
checkpoints и patches
Ignis и Markdown vault
.agents
```

Web, Android и iOS — клиенты управления. Они не требуют локального Pi, Git или доступа к локальной файловой системе.

Не входят в текущий scope:

```text
локальный worker
синхронизация с локальным checkout
Pi CLI
handoff между Web и CLI
```

---

## 1.2. Одна PiSession на один Chat

Фиксируем вариант A:

```text
Chat
└── PiSession
    ├── Task 1
    ├── Task 2
    └── Task 3
```

Правила:

* Chat владеет одной постоянной PiSession.
* Task не создаёт отдельную PiSession.
* Все последовательные задачи одного Chat продолжают одну историю Pi.
* Task хранит диапазон session entries, относящийся к ней.
* В один момент у PiSession может быть только один writer runtime.
* В обычном implementation chat может быть только одна активная writable Task.
* Исторических, merged, cancelled и rollback-задач в Chat может быть несколько.

Минимальная связь сущностей:

```ts
type Chat = {
  id: string;
  projectId: string;
  piSessionId: string;
  activeTaskId: string | null;
  activeLeafEntryId: string | null;
};

type Task = {
  id: string;
  chatId: string;
  branchName: string;
  worktreePath: string;
  baseSha: string;
  currentHeadSha: string;
  startPiEntryId: string | null;
  endPiEntryId: string | null;
  status: TaskStatus;
};
```

---

## 1.3. Переключение PiSession между репо и worktree

Одна PiSession может последовательно работать в разных filesystem contexts.

Когда у Chat нет активной writable Task:

```text
PiSession cwd = основное репо
tools = read-only
```

Когда создаётся writable Task:

```text
Pi runtime текущего Chat останавливается
создаётся worktree
PiSession повторно открывается
cwd = worktree Task
write tools включаются
```

После merge или отмены Task:

```text
runtime Task закрывается
PiSession остаётся
следующий runtime может снова открыться на основном репо
или на worktree следующей Task
```

При каждой смене filesystem context в историю добавляется служебный marker:

```text
workspace_context_changed

previousTaskId
newTaskId
previousCwd
newCwd
baseSha
targetBranch
lastMergeCommit
reason
```

Перед первой работой в новом worktree агент обязан заново проверить:

```text
git status
текущий SHA
файлы проекта
AGENTS.md
.agents
diff относительно baseSha
```

Это снижает риск того, что Pi продолжит рассуждать о файловом состоянии предыдущей Task.

# 2. Chat, Task и orchestration

## 2.1. Обычный Chat

Обычный Chat может последовательно иметь несколько Tasks:

```text
Chat A
├── Task A1 — merged
├── Task A2 — cancelled
└── Task A3 — active
```

Но одновременно активна только одна writable Task.

После merge пользователь может продолжить писать в том же Chat. При следующем запросе на изменение файлов создаётся новая Task и новый worktree от актуального состояния основного репо.

## 2.2. Discussion и planning

Discussion и planning:

* используют ту же PiSession Chat;
* читают основное репо;
* не создают worktree;
* не получают write tools;
* не прикрепляются автоматически к чужой Task.

Чтобы работать в контексте конкретной Task, пользователь открывает Chat, к которому эта Task привязана.

## 2.3. Orchestration Chat

Чтобы одна PiSession не пыталась одновременно работать в нескольких worktree, фиксируем:

* orchestration Chat управляет Tasks на уровне приложения;
* он не исполняет writable tool calls внутри worktree дочерних Tasks;
* каждая writable Task привязана к своему implementation Chat;
* orchestration Chat может создавать, наблюдать, останавливать, возобновлять, проверять и предлагать merge этих Tasks;
* для детальной работы пользователь переключается в соответствующий Chat.

Пример:

```text
Orchestration Chat
├── контролирует Task A в Chat A
├── контролирует Task B в Chat B
└── контролирует Task C в Chat C
```

Если orchestration Chat создаёт новую реализационную работу, backend создаёт или выбирает implementation Chat и уже к нему привязывает Task.

# 3. Что является шагом работы

Шаг фиксируется так:

> Один шаг начинается с отправки сообщения пользователем и заканчивается terminal-событием работы агента.

```text
user message
→ agent_start
→ text/tool calls
→ agent_end
= один завершённый шаг
```

## 3.1. Steer

Steer относится к уже выполняющемуся шагу и не создаёт отдельный checkpoint сам по себе.

```text
user message
→ agent работает
→ steer
→ agent продолжает
→ agent_end
→ один checkpoint
```

## 3.2. Follow-up

Follow-up становится следующим шагом после завершения текущего:

```text
Step 1
  user message
  agent work
  agent_end
  checkpoint 1

Step 2
  follow-up message
  agent work
  agent_end
  checkpoint 2
```

## 3.3. Быстрые действия

Action или skill, который запускает работу агента, должен быть представлен в истории как пользовательский или служебный executable message. Тогда он подчиняется той же модели шага и checkpoint.

---

# 4. Checkpoint после каждого завершённого шага

После каждого успешного завершённого шага создаётся checkpoint.

Checkpoint создаётся даже тогда, когда файлы не изменились.

```ts
type Checkpoint = {
  id: string;
  chatId: string;
  taskId: string;
  runId: string;
  stepNumber: number;

  piEntryId: string;
  beforeSha: string;
  afterSha: string;

  hasFileChanges: boolean;
  changedFiles: number;
  patchPath: string | null;

  createdAt: string;
};
```

## 4.1. Если файлы изменились

```text
git add -A
внутренний commit в task branch
beforeSha != afterSha
worktree становится clean
patch сохраняется вне worktree
```

## 4.2. Если файлы не изменились

```text
checkpoint record создаётся
beforeSha == afterSha
пустой Git commit не создаётся
Pi entry сохраняется
```

Так можно откатывать не только файловое состояние, но и контекст разговора.

## 4.3. Aborted и failed Run

Aborted или failed Run не считается завершённым шагом.

По умолчанию:

* автоматический стабильный checkpoint не создаётся;
* изменения остаются в worktree;
* Task получает `paused_dirty` или `paused_clean`;
* пользователь может явно создать ручной checkpoint.

# 5. Abort, продолжение и отмена Task

## 5.1. Abort Run

Abort:

* останавливает активный Pi run;
* завершает дочернюю process group;
* не удаляет изменения;
* не выполняет Git reset;
* не удаляет queued messages;
* переводит Task в paused-состояние.

После Abort доступны действия:

```text
Продолжить с текущими изменениями
Отправить корректировку
Создать checkpoint текущего состояния
Откатить worktree
Отменить Task
```

## 5.2. Продолжение после Abort

Новое сообщение запускает новый Run в том же worktree.

Перед продолжением агент получает recovery context:

```text
Предыдущий Run был остановлен.
Проверь:
- git status;
- diff от последнего checkpoint;
- незавершённые команды;
- созданные и удалённые файлы;
- соответствие текущего состояния задаче.
После проверки продолжи по новому сообщению пользователя.
```

## 5.3. Отмена Task

Оставляем два явных варианта.

### Отменить и сохранить

```text
Run останавливается
очередь очищается
Task → cancelled_archived
branch сохраняется
worktree сохраняется или архивируется
изменения можно посмотреть позже
```

### Отменить и удалить изменения

```text
Run останавливается
очередь очищается
пользователь подтверждает destructive action
worktree удаляется
служебная task branch удаляется
Task → cancelled_discarded
audit metadata сохраняется
```

Обычный Abort никогда не выполняет поведение `cancelled_discarded`.

# 6. Rollback и Fork

С учётом одной PiSession на Chat эти действия различаются.

## 6.1. Rollback

Rollback создаёт новую Task внутри того же Chat.

```text
Chat A / PiSession A
├── Task A1
│   ├── C1
│   ├── C2
│   └── C3
└── rollback from C2
    └── Task A2
```

При rollback от C2:

* исходная Task A1 сохраняется;
* создаётся Task A2;
* создаётся новая branch;
* создаётся новый worktree от `C2.afterSha`;
* active leaf PiSession переводится на `C2.piEntryId`;
* дальнейшая история Pi образует новую ветку;
* Task A2 становится активной Task Chat.

Переключение между линиями не должно переписывать один worktree. Каждая Task имеет собственный worktree. Пользовательский интерфейс просто выбирает соответствующую Task и соответствующую ветку PiSession.

Rollback допускается только если:

* нет активного Run;
* выбран стабильный checkpoint;
* текущая Task остановлена, завершена или переведена в неактивное состояние.

## 6.2. Fork

Fork создаёт отдельный Chat и отдельную PiSession.

```text
Chat A / PiSession A / Task A
              │
              └── fork from C2
                    ↓
Chat B / PiSession B / Task B
```

Fork создаёт:

```text
новый Chat
новую PiSession, ответвлённую от piEntryId checkpoint
новую Task
новую Git-ветку
новый worktree от afterSha checkpoint
```

Разница:

```text
Rollback:
  новая Task в том же Chat и в дереве той же PiSession

Fork:
  новый Chat и новая независимая PiSession
```

# 7. Основное репо и remote

## 7.1. Никакой автоматической синхронизации

Не выполняются автоматически:

```text
git fetch
git pull
git push
rebase от remote
переключение веток
```

Task создаётся от текущего состояния target branch в основном репо на VPS.

Если remote изменился, приложение не узнаёт об этом до явного действия пользователя.

## 7.2. Явное действие «Синхронизировать»

На уровне Project доступна явная операция:

```text
Синхронизировать с remote
```

Минимальный безопасный workflow:

```text
1. Проверить, что основное репо clean.
2. Выполнить git fetch.
3. Сравнить local target и remote target.
4. Если возможен fast-forward — предложить применить его.
5. Если ветки разошлись — ничего не переписывать автоматически.
6. После изменения target HEAD пересчитать stale для Tasks.
```

## 7.3. Merge и push — разные операции

По умолчанию:

```text
Merge Task
  → создаёт squash commit в основном репо на VPS

Push to remote
  → отдельное явное действие
```

Merge не должен скрыто выполнять `git push`.

Это позволяет сначала проверить состояние VPS-репозитория, а потом отдельно решить, отправлять ли его в remote.

## 7.4. Branch UI

Пока отсутствуют:

```text
branch picker
checkout произвольной ветки
создание пользовательской ветки
force push
UI для сложного разрешения divergence основного repo
```

Target branch задаётся конфигурацией Project.

# 8. Поведение после рестарта backend

После рестарта никакая работа не продолжается автоматически.

Для всех выполнявшихся Tasks:

```text
Run → interrupted
Task → paused_after_restart
queue → сохранена, но paused
Pi process → не восстанавливается автоматически
```

После открытия Chat пользователь видит:

```text
Работа была прервана перезапуском backend.
Очередь сохранена.
Для продолжения отправьте новое сообщение.
```

Новое сообщение создаёт новый Run.

Перед выполнением агент обязан проверить:

```text
git status
текущий HEAD
diff от последнего checkpoint
незавершённые изменения
оставшиеся процессы
соответствие worktree Task
```

Только после этой проверки он продолжает пользовательскую задачу.

Queued follow-ups сами не запускаются. Пользователь может:

```text
оставить очередь
изменить порядок
удалить элементы
очистить очередь
отправить новое сообщение для продолжения
```

# 9. Ignis и Obsidian

Ignis доступен:

```text
Web
Android
iOS
```

На mobile он открывается через WebView или встроенный web route и работает с vault на VPS.

Фиксируем упрощённую модель конкурентности:

* агент может менять Markdown-файлы;
* Ignis может менять Markdown-файлы;
* пользователь не редактирует vault во время активной работы Task;
* специальный механизм конкурентного merge Ignis ↔ агент пока не создаётся;
* optimistic file locking и hash conflict detection не являются критерием текущего этапа.

Практически UI должен показывать, что Task активна. Но блокировать Ignis технически не обязательно.

После окончания Task пользователь может открыть Ignis и увидеть актуальное состояние файлов.

Если пользователь нарушает договорённость и редактирует тот же файл во время Run, корректное автоматическое разрешение такого конфликта в текущую версию не входит.

# 10. Sandbox и процессы

Конкретная технология может быть выбрана при реализации:

```text
container per Task
или
отдельный process supervisor per Task
```

Но функциональные требования обязательны:

* отдельная process group;
* cwd строго равен task worktree;
* Abort останавливает весь процессный subtree;
* timeout;
* ограничение output;
* ограниченный набор secrets;
* запрет случайного доступа к worktree другой Task;
* cleanup после падения backend;
* audit запускаемых процессов.

Отдельный plugin sandbox пока не требуется.

Поэтому пока не реализуются:

```text
login

access tokens
```

```text

публичный интернет endpoint не используется
```

Это ограничение должно быть явно записано в README и deployment-документации.

# 14. Уточнённые критерии приёмки

## 14.1. Одна session и несколько последовательных Tasks

Проверка:

1. Создать Chat.
2. Выполнить discussion.
3. Создать Task A.
4. Изменить файлы.
5. Слить Task A.
6. Продолжить разговор.
7. Создать Task B.
8. Изменить другие файлы.

Работа принята, если:

* Chat всё время использует одну PiSession;
* Task A и B имеют разные worktrees;
* Pi помнит разговор до Task A;
* Pi помнит результат Task A при работе над Task B;
* перед Task B runtime открыл ту же session с новым cwd;
* изменения Task A не остались незакоммиченными в worktree B.

## 14.2. Checkpoint каждого шага

Проверка:

1. Отправить сообщение без файловых изменений.
2. Получить ответ.
3. Отправить сообщение с файловыми изменениями.
4. Получить ответ.

Работа принята, если:

* созданы два checkpoint records;
* первый имеет одинаковые `beforeSha` и `afterSha`;
* второй имеет разные SHA;
* пустой Git commit для первого не создан;
* оба checkpoint связаны со своими Pi entries.

## 14.3. Abort без потери файлов

Проверка:

1. Агент изменяет файлы.
2. Выполняется Abort до `agent_end`.
3. Открывается diff.
4. Отправляется корректирующее сообщение.

Работа принята, если:

* файлы остались изменёнными;
* автоматический reset не выполнялся;
* Task получила paused-состояние;
* новый Run увидел предыдущие изменения;
* пользователь может отдельно нажать revert или cancel-and-discard.

## 14.4. Rollback внутри одного Chat

Проверка:

1. Создать C1, C2 и C3.
2. Выполнить rollback от C2.
3. Создать Task B.
4. Продолжить работу.
5. Вернуться к истории Task A.

Работа принята, если:

* Task B находится в том же Chat;
* используется та же PiSession;
* активная ветка Pi начинается от C2;
* worktree B начинается от SHA C2;
* Task A и её дальнейшая история не удалены;
* worktree A не был переписан.

## 14.5. Fork в новый Chat

Проверка:

1. Выбрать C2 в Chat A.
2. Выполнить Fork.
3. Продолжить Chat A и Chat B по-разному.

Работа принята, если:

* Chat B имеет отдельную PiSession;
* история до C2 совпадает;
* история после C2 независима;
* Tasks используют разные worktrees;
* изменения не смешиваются.

## 14.6. Явная remote sync

Проверка:

1. Изменить remote target.
2. Не нажимать Sync.
3. Убедиться, что VPS repo не изменился.
4. Нажать Sync.
5. Выполнить fast-forward.
6. Проверить существующие Tasks.

Работа принята, если:

* фонового fetch/pull не было;
* repo изменился только после явного действия;
* Tasks от старого SHA получили stale;
* никакая активная Task не была автоматически rebase;
* merge не выполнил автоматический push.

## 14.7. Restart с paused queue

Проверка:

1. Запустить Run.
2. Добавить follow-up.
3. Перезапустить backend.
4. Подождать.
5. Открыть Chat.
6. Отправить новое сообщение.

Работа принята, если:

* follow-up не запустился автоматически;
* очередь сохранилась;
* старый Run стал interrupted;
* Task стала paused_after_restart;
* новый Run сначала проверил worktree;
* существующие изменения не потерялись.

## 14.8. Orchestration

Проверка:

1. Создать orchestration Chat.
2. Создать через него две работы.
3. Получить два implementation Chats и две Tasks.
4. Запустить их параллельно.
5. Наблюдать их из orchestration Chat.
6. Открыть каждый implementation Chat.

Работа принята, если:

* orchestration PiSession не выполняет запись в worktrees дочерних Tasks;
* каждая Task привязана к своему Chat;
* каждая Task имеет собственный worktree;
* пользователь переходит в нужный контекст переключением Chat;
* управление одной Task не влияет на другую.

---

# 15. Что осталось решить только на уровне реализации

Критических продуктовых неоднозначностей для основного контура больше нет. Остаются технические решения, которые не меняют пользовательскую модель:

* container или process supervisor для первой версии;
* формат служебного `workspace_context_changed`;
* naming внутренних checkpoint commits;
* срок хранения отменённых worktrees;
* механизм runtime generation для отбрасывания поздних событий старого Pi process;
* реализация Pi session fork и переключения active leaf;
* отдельные названия действий `Sync from remote` и `Push to remote`;
* способ встраивания Ignis в React Native WebView.

# 16. Следующий реализационный milestone

Следующий вертикальный срез должен доказать именно принятую модель:

```text
1. Создать Chat и одну постоянную PiSession.
2. Провести discussion без worktree.
3. Создать writable Task и worktree.
4. Переоткрыть ту же PiSession с cwd worktree.
5. Выполнить два пользовательских шага.
6. Создать два checkpoint.
7. Остановить третий шаг через Abort без отката файлов.
8. Продолжить новым сообщением.
9. Выполнить squash merge одним commit.
10. Создать следующую Task в том же Chat.
11. Переоткрыть ту же PiSession в новом worktree.
12. Перезапустить backend и подтвердить paused recovery.
```

После прохождения этого сценария можно переходить к rollback, fork, orchestration и параллельным Tasks.
